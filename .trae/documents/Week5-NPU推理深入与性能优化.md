#### 答案

**1. CANN软件栈分为哪几层？各层的职责是什么？**

| 层级 | 职责 |
|------|------|
| **应用层** | 深度学习框架 |
| **框架适配层** | 框架到CANN的适配 |
| **计算图引擎层(GE)** | 图优化、内存分配、任务调度 |
| **算子编译层(TBE)** | 算子开发、编译 |
| **运行时层(ACL)** | 模型加载、内存管理、任务执行 |
| **硬件层** | Ascend NPU |

**2. ACL推理的完整流程是什么？**

流程：
1. 初始化: aclInit(), aclrtSetDevice()
2. 加载模型: aclmdlLoadFromFile()
3. 准备输入输出: aclrtMalloc(), aclCreateDataBuffer()
4. 执行推理: aclrtMemcpy(), aclmdlExecute()
5. 释放资源: aclmdlUnload(), aclFinalize()

**3. GE的图优化有哪些策略？**

优化策略：
- **常量折叠**: 预计算常量表达式
- **算子融合**: 将多个算子融合为一个
- **死代码消除**: 移除无效计算
- **内存优化**: 复用内存，减少占用
- **并行优化**: 识别可并行的算子

**4. TBE的作用是什么？如何开发自定义算子？**

TBE作用：
- 提供算子开发框架
- 支持TIK和Davinci C++两种开发方式
- 编译算子到NPU可执行格式

开发步骤：
1. 定义算子逻辑 (TIK DSL)
2. 实现算子计算
3. 编译算子
4. 注册算子到框架

**5. 请对比CANN和CUDA的软件栈。**

| 层级 | CANN | CUDA |
|------|------|------|
| 应用层 | PyTorch/TF | PyTorch/TF |
| 框架适配 | Torch_Ascend | Torch_CUDA |
| 计算图引擎 | GE | XLA/TensorRT |
| 算子编译 | TBE | NVCC |
| 运行时 | ACL | CUDA Runtime |
| 硬件 | Ascend NPU | NVIDIA GPU |

#### Offline vs Online模式

```
Offline模式 (离线推理):
┌─────────────────────────────────────────────────────────┐
│ 特点:                                                   │
│ - 模型提前编译为离线模型 (.om文件)                       │
│ - 推理时直接加载.om文件                                 │
│ - 启动快，推理效率高                                     │
│ - 模型固定，无法动态修改                                 │
│                                                         │
│ 适用场景:                                               │
│ - 生产环境部署                                          │
│ - 模型固定不变                                          │
│ - 追求最高性能                                          │
└─────────────────────────────────────────────────────────┘

Online模式 (在线推理):
┌─────────────────────────────────────────────────────────┐
│ 特点:                                                   │
│ - 推理时动态编译模型                                     │
│ - 支持动态Shape                                         │
│ - 灵活性高，开发方便                                     │
│ - 首次推理需要编译时间                                   │
│                                                         │
│ 适用场景:                                               │
│ - 开发调试                                              │
│ - 模型频繁变化                                          │
│ - 需要动态Shape                                         │
└─────────────────────────────────────────────────────────┘
```

#### Offline模式开发流程

```
Offline模式开发:

Step 1: 模型转换
┌─────────────────────────────────────────────────────────┐
│ 使用ATC工具转换模型:                                     │
│ atc --model=model.onnx \                                │
│      --framework=5 \                                    │
│      --output=model \                                   │
│      --soc_version=Ascend910B                           │
│                                                         │
│ 输出: model.om (离线模型文件)                            │
└─────────────────────────────────────────────────────────┘
    ↓
Step 2: 加载模型
┌─────────────────────────────────────────────────────────┐
│ aclmdlLoadFromFile("model.om", &model_id)               │
└─────────────────────────────────────────────────────────┘
    ↓
Step 3: 执行推理
┌─────────────────────────────────────────────────────────┐
│ aclmdlExecute(model_id, input_dataset, output_dataset)  │
└─────────────────────────────────────────────────────────┘
```

#### Online模式开发流程

```
Online模式开发:

Step 1: 加载原始模型
┌─────────────────────────────────────────────────────────┐

┌─────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────┘
    ↓
Step 3: 后续推理
┌─────────────────────────────────────────────────────────┐
│ output = model(input)  // 使用编译后的模型              │
└─────────────────────────────────────────────────────────┘
```

#### 动态Shape处理

```
动态Shape支持:

方式1: 动态Batch
┌─────────────────────────────────────────────────────────┐
│ 输入Shape: [-1, 3, 224, 224]  // -1表示动态维度         │

┌─────────────────────────────────────────────────────────┐

方式3: 完全动态
┌─────────────────────────────────────────────────────────┐

```

**5. 请设计一个支持动态Batch的推理服务。**

```python
class DynamicBatchInferenceService:
    def __init__(self, om_model_path):
        self.model_id = self._load_model(om_model_path)
        self.max_batch_size = 32
    
    def infer(self, inputs):
        batch_size = len(inputs)
        if batch_size > self.max_batch_size:
            # 分批处理
            results = []
            for i in range(0, batch_size, self.max_batch_size):
                batch = inputs[i:i+self.max_batch_size]
                results.extend(self._infer_batch(batch))
            return results
        else:
            return self._infer_batch(inputs)
    
    def _infer_batch(self, batch):
        # 执行推理
        return aclmdlExecute(self.model_id, batch)
```

#### TIK算子示例

```python
import tbe.dsl as tbe

def add_compute(x, y, z, kernel_name="add"):
    """
    实现两个张量的逐元素相加
    """
    # 计算逻辑
    res = tbe.vadd(x, y)
    
    return res

def add(x, y, z, kernel_name="add"):
    """
    算子入口函数
    """
    # 1. 定义输入输出tensor
    data_x = tbe.placeholder(x.shape, name="data_x", dtype=x.dtype)
    data_y = tbe.placeholder(y.shape, name="data_y", dtype=y.dtype)
    
    # 2. 调用计算函数
    res = add_compute(data_x, data_y, z)
    
    # 3. 调度和编译
    with tbe.target_scope():
        sch = tbe.auto_schedule(res)
    
    config = {"name": kernel_name, "tensor_list": [data_x, data_y, res]}
    tbe.cce_build_code(sch, config)
```

### 核心知识点

#### 量化基本原理

```
量化: 将高精度浮点数映射到低精度整数

FP32 → INT8:
┌─────────────────────────────────────────────────────────┐
│ 原始范围: [-3.4e38, 3.4e38]                             │
│ 量化范围: [-128, 127]                                   │
│                                                         │
│ 量化公式:                                                │
│ q = round(x / scale) + zero_point                      │
│                                                         │
│ 反量化:                                                  │
│ x = (q - zero_point) * scale                           │
└─────────────────────────────────────────────────────────┘
```

#### 量化方法对比

| 方法 | 描述 | 精度损失 | 计算量 |
|------|------|----------|--------|
| **PTQ** | 训练后量化 | 较大 | 无需训练 |
| **QAT** | 量化感知训练 | 较小 | 需要训练 |
| **GPTQ** | 基于Hessian的PTQ | 小 | 需要校准数据 |
| **AWQ** | 激活感知量化 | 小 | 需要校准数据 |

#### INT8量化推理

```
INT8矩阵乘法:

FP16: Y = W * X
INT8: Y = (W_int8 * X_int8) * scale_W * scale_X

优势:
- 内存占用减少50%
- 计算速度提升 (INT8 Tensor Core)
- 带宽需求减少

挑战:
- 精度损失
- 动态范围处理
- 异常值处理
```

#### FP8量化

```
FP8格式:
┌─────────────────────────────────────────────────────────┐
│ E4M3: 1符号位 + 4指数位 + 3尾数位                        │
│ E5M2: 1符号位 + 5指数位 + 2尾数位                        │
│                                                         │
│ E4M3: 更高精度，适合权重                                 │
│ E5M2: 更大范围，适合激活                                 │
└─────────────────────────────────────────────────────────┘

优势:
- 硬件原生支持 (H100)
- 精度比INT8高
- 训练和推理统一
```

#### KV Cache量化

```
KV Cache量化:

原始: FP16 KV Cache
量化: INT8/FP8 KV Cache

优势:
- 内存占用减少50%
- 支持更长序列
- 支持更大batch

实现:
- 按head量化
- 按token量化
- 动态量化
```

#### 二、简答题

**1. LLM推理完整流程**

流程：
1. 输入处理: Prompt → Tokenizer → Token IDs
2. Prefill阶段: 并行处理整个Prompt，计算KV Cache，输出第一个token
3. Decode阶段: 自回归生成后续token，复用KV Cache
4. 输出处理: Token IDs → Detokenizer → 文本

Prefill vs Decode区别：
- Prefill: 计算密集型，并行处理
- Decode: 内存带宽密集型，串行处理

**2. PagedAttention核心思想**

核心思想：
- 借鉴操作系统的虚拟内存管理
- 将KV Cache划分为固定大小的Block
- 按需分配，逻辑地址映射到物理地址

KV Cache共享实现：
- Block可被多个请求引用
- 引用计数管理
- Copy-on-Write机制

**3. vLLM调度器**

队列作用：
- waiting: 存储新到达的请求
- running: 存储正在执行的请求
- swapped: 存储被换出的请求

调度流程：
1. 处理swapped队列
2. 调度running队列 (Decode)
3. 调度waiting队列 (Prefill)
4. 生成SchedulerOutput

**4. CANN软件栈**

分层：
- 应用层: PyTorch/TF
- 框架适配层: Torch_Ascend
- 计算图引擎层: GE
- 算子编译层: TBE
- 运行时层: ACL
- 硬件层: Ascend NPU

ACL推理流程：
1. 初始化: aclInit(), aclrtSetDevice()
2. 加载模型: aclmdlLoadFromFile()
3. 准备输入输出
4. 执行推理
5. 释放资源
